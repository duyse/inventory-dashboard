# 📦 Inventory Dashboard

A modern, fast React + TypeScript inventory dashboard. It uses a mock backend for development and testing, allowing for a seamless frontend development experience.

## 🛠️ Tech Stack
* **Framework:** React + Vite
* **Language:** TypeScript (Strict Mode)
* **Data Fetching & State:** TanStack Query
* **Styling:** Tailwind CSS
* **Testing:** Jest + `ts-jest`
* **API Mocking:** Express (Backend) + MSW (Browser)

---

## 🚀 Getting Started

### Prerequisites
* Node.js 20 or higher
* A modern web browser

### 1. Install Dependencies
```bash
npm install
```

### 2. Run the Development Server
Start the app with hot-module reloading. API requests will be automatically intercepted and mocked by MSW.
```bash
npm run dev
```
*Vite will provide a local URL (usually `http://localhost:5174`). Open it in your browser.*

### 3. Build for Production
Type-check the project and generate a production-ready build in the `dist/` folder:
```bash
npm run build
```
To preview this production build locally:
```bash
npm run build && npm run preview
```

### 4. Run Tests & Linting
We use Jest for testing the API client, mock server, and utilities.
```bash
# Run all tests
npm test

# Run a specific test file
npm test -- tests/api.test.ts

# Run the linter
npm run lint
```

## 📁 Project Structure

* **`src/App.tsx`** – The root React component.
* **`src/components/`** – UI components.
* **`src/api.ts`** – Strongly-typed API client.
* **`src/server.ts`** – Express mock backend.
* **`src/mocks/`** – MSW handlers for intercepting browser requests.
* **`tests/`** – Jest test suites.
* **`system-design.md`** – Notes on the system architecture.
* **`build.md`** – Notes on the build process and data models.

## 🤖 AI Collaboration Methodology

This project was built in close collaboration with Gemini 3.1 Pro. To ensure the final codebase remained high-quality and production-ready, I treated the AI as a fast but literal assistant, requiring strict human oversight. 

Here is how I managed the workflow:

### 1. Guiding the AI
* **Design before coding:** I had the AI map out the data models, components, and API routes in `build.md` first. Locking in the design meant we didn't have to guess how things fit together later.
* **Feature by feature (Vertical Slices):** I asked the AI to build one complete feature at a time—from the database mock to the UI component and its test. This kept code changes small and easy to review.
* **Upfront constraints:** I included strict rules in every prompt (e.g., *must use strict TypeScript, no `any` types, use Tailwind*). Setting rules early prevented the AI from drifting off-course.
* **Show, don't just tell:** Instead of explaining what I wanted, I pasted existing code snippets as examples. The AI adapts to concrete patterns much better than abstract instructions.

### 2. Verifying the Output
I never blindly accepted AI-generated code. My review loop was:
* **Read every line:** I reviewed the entire code diff, not just the summary, to catch sneaky or unrelated changes.
* **Test the logic, not just the types:** I ran `npm run build` and `npm test` after every change. Since code can compile perfectly but still fail in practice, I also manually clicked through the UI to test edge cases (like loading or error states).
* **Reject over-engineering:** If the AI added unnecessary features, overly complex abstractions, or defensive code for impossible errors, I rejected the changes and asked for the simplest solution.
* **Blind AI reviews:** For tricky logic (like sorting or filtering), I opened a *new* chat without the previous context and asked the AI to review its own code. This helped catch mistakes the AI had previously convinced itself were correct.

### 3. Enforcing Quality
* **Strict typing:** The API boundaries are strictly typed in `src/types.ts` and shared across the app. If the backend and frontend don't match, the app won't compile.
* **Targeted testing:** I focused tests on the areas most likely to break (API clients, server routes, utility functions) rather than writing fragile snapshot tests for UI components.
* **Zero dead code:** Any "just in case" code generated by the AI—such as unused variables or unnecessary comments explaining *what* the code does—was deleted. Comments were only kept if they explained *why* a specific technical decision was made.
