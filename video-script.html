Hi, I'm Duy Nguyen. Thanks for taking the time to review my submission.

I picked Scenario B — the Intelligent Inventory Dashboard. The job in plain English: dealership managers need one screen that shows every car on their lot, lets them filter it, makes anything sitting longer than ninety days stand out, and lets them write down what they plan to do about it — and that note has to still be there tomorrow.

So at its core it's a small CRUD app with one interesting business rule and one interesting filter surface. I designed it to match that, not to look like a system five times its size. In the next few minutes I'll walk you through the design, the implementation, how I worked with AI on it, a quick demo, and what I learned.

2. System Design Walkthrough (0:45 – 3:00)

Screen share: system-design.md + architecture diagram.

Assumptions first, because the brief is intentionally vague:

  • Multi-dealership from day one — every vehicle is scoped to a dealershipId. Auth isn't built, but the data model is ready for it.
  • Age = now − received_at. The 90-day threshold is a config value, not a magic number.
  • Inventory is read-heavy — managers view far more than vehicles arrive. Optimise for fast reads, simple writes.
  • One current status per vehicle, plus history — append-only audit log alongside a denormalised current-status column.
  • Realistic scale — hundreds to a few thousand vehicles per dealership. Not a web-scale problem.

Architecture: a stateless Express + TypeScript API in front of one PostgreSQL database, with Pino structured logs and prom-client RED metrics. When traffic grows I run more API copies behind a load balancer; Postgres scales vertically and later with read replicas.

The key design call is computing isAging on read, not storing it. It's always correct, no nightly cron job, and the cost is one subtraction per row.

Writes happen in a single transaction — insert into the immutable vehicle_actions history, update the denormalised current status on vehicles. History gives me audit; the column keeps the dashboard query a single SELECT with no join. Indexes are on (dealership_id, received_at) and (dealership_id, make, model) — the two queries that actually matter.

Things I deliberately did not pick: microservices, a message queue, Redis, GraphQL, Kubernetes. Each one would be operational tax with no benefit at this size. I'd rather start small and add things when a metric tells me to.

3. Implementation Highlights (3:00 – 4:15)

Screen share: VS Code — show src/types.ts, src/api.ts, src/server.ts, a component file.

The submission is a React + Vite + TypeScript SPA in strict mode, styled with Tailwind, data-fetched with TanStack Query, and backed by an Express mock backend plus MSW for browser interception. Tests are Jest with ts-jest.

A few things I want to call out:

  • Strictly typed API boundary — Vehicle, status enum, and request/response shapes live in src/types.ts and are shared by the client, the mock server, and the tests. If the contract drifts, the app simply doesn't compile.
  • TanStack Query handles loading, error, and refetch after a status update, so the UI stays in sync without me hand-rolling cache logic.
  • Aging logic in one place — the mock server computes isAging, exactly like the real API would. The frontend just renders the flag.
  • Tests are targeted, not ceremonial — API client, mock routes, and the filter/age utilities. No fragile UI snapshot tests.

4. AI Collaboration Story (4:15 – 5:30)

Back to camera, or split screen with README's "AI Collaboration" section.

I treated AI as a fast but literal pair, never as the author. My loop had three rules.

Design before code. I wrote the problem in my own words first, in build.md and system-design.md, before letting an AI talk at me. Then I asked for critique, not solutions — prompts like "here's my data model, what am I missing?" That surfaced things like timezone questions on received_at.

Generate wide, then cut. I asked for every observability signal you'd want for a system like this — and then deleted about seventy percent. Picking what not to include was my job.

Verify everything. Every line of generated code got read in full, type-checked, tested, and clicked through in the browser. Where I overruled the AI: it suggested a materialised view for aging stock, microservices, and OpenTelemetry from day one. I rejected all three with reasons documented in section 13 of the design doc. The engineering judgement — what to throw away — is the part that has my name on it.

5. Application Demo (5:30 – 7:30)

Here's the dashboard for dealership 42. The filter bar at the top lets me narrow by make, model, and a "show aging stock only" toggle.

The table shows VIN, make, model, year, price, days in inventory, and current status. Notice the amber-flagged rows — those are the aging ones, more than ninety days on the lot. That's isAging coming straight from the API.

Let me flip the "aging only" toggle — the list narrows to just the cars that need attention.

Now I'll click Edit Action on this aging row. The modal opens with a status dropdown — Price Reduction Planned, Send to Auction, In Repair — and a note field. I'll set it to Price Reduction Planned, add the note "Drop price by 5%", and submit.

The PATCH fires, TanStack Query refetches, and you can see the row's status update live. If I refresh the page, the note is still there — that's the persistence requirement satisfied.

6. What I Learned & Challenges (7:30 – 9:00)

On camera. Closing slide.

Three things stuck with me.

The hardest part of the design wasn't what to add — it was what to leave out. The AI was happy to suggest Redis, Kafka, microservices, OpenTelemetry. Resisting that, and writing down honest reasons against each, was more valuable than any of those tools would have been at this size.

Computing isAging on read is a small decision that pays back constantly. No cron, no staleness, no "why is this car still flagged" support tickets. Boring is good.

Challenges: getting MSW and the Express mock server to agree on the exact same response shape was fiddlier than I expected — I solved it by making both consume the same TypeScript types from src/types.ts, so any divergence becomes a compile error. The other one was discipline with AI output: it's easy to accept a clean-looking diff that quietly adds defensive code for impossible errors. I rejected those passes and asked for the simplest version every time.

If I were taking this past v1, the next steps are real auth, rate limiting on the action endpoint, and distributed tracing the moment a second service shows up. Thanks for watching.
